채팅이나 실시간 알림 기능을 만들려고 하면 자주 등장하는 기술이 있다.

- WebSocket
- STOMP
- RSocket

처음 보면 모두 “실시간 통신”과 관련된 기술처럼 보이기 때문에 비슷한 레벨의 기술이라고 생각하기 쉽다. 하지만 실제로는 각 기술이 담당하는 역할과 계층이 다르다.

이 글에서는 WebSocket, STOMP, RSocket이 각각 무엇인지, 서로 어떤 관계인지, 그리고 채팅 서비스를 만들 때 어떤 선택이 현실적인지 정리해보려고 한다.

---

## 1. 먼저 계층부터 구분해보자

세 기술의 관계를 단순화하면 다음과 같다.

```text
TCP
 ├─ WebSocket
 │   ├─ STOMP
 │   └─ RSocket over WebSocket
 │
 └─ RSocket over TCP
```

핵심은 다음과 같다.

```text
WebSocket = 양방향 통신을 위한 연결 프로토콜

STOMP = WebSocket 위에서 사용하는 메시징 규칙

RSocket = TCP 또는 WebSocket 위에서 동작하는 애플리케이션 메시징 프로토콜
```

즉, WebSocket과 STOMP/RSocket은 같은 레벨의 기술이 아니다.

WebSocket은 클라이언트와 서버가 양방향으로 메시지를 주고받을 수 있게 해주는 통신 통로에 가깝고, STOMP와 RSocket은 그 위에서 메시지를 어떤 방식으로 주고받을지 정의하는 상위 프로토콜에 가깝다.

---

## 2. WebSocket이란?

WebSocket은 클라이언트와 서버가 하나의 연결을 유지한 상태에서 양방향으로 메시지를 주고받을 수 있게 해주는 프로토콜이다.

일반적인 HTTP 통신은 기본적으로 요청과 응답 구조다.

```text
클라이언트 요청 → 서버 응답
클라이언트 요청 → 서버 응답
클라이언트 요청 → 서버 응답
```

반면 WebSocket은 한 번 연결을 맺으면 연결을 유지한 상태에서 양쪽이 자유롭게 메시지를 주고받을 수 있다.

```text
클라이언트 ↔ 서버
클라이언트 ↔ 서버
클라이언트 ↔ 서버
```

이런 특성 때문에 WebSocket은 다음과 같은 기능에 자주 사용된다.

- 채팅
    
- 실시간 알림
    
- 주식 시세
    
- 실시간 게임
    
- 협업 편집
    
- 실시간 모니터링
    

하지만 WebSocket 자체는 어디까지나 “양방향 통신 통로”에 가깝다. WebSocket을 사용한다고 해서 채팅에 필요한 모든 기능이 자동으로 해결되는 것은 아니다.

예를 들어 아래 기능들은 WebSocket이 직접 제공해주는 것이 아니다.

```text
- 채팅방 구독
- 사용자별 메시지 라우팅
- 메시지 ACK
- 메시지 재전송
- 중복 메시지 방지
- 메시지 순서 보장
- 오프라인 메시지 처리
- 재연결 시 누락 메시지 복구
```

이런 부분은 애플리케이션에서 직접 설계하거나, STOMP나 RSocket 같은 상위 프로토콜을 활용해 보완해야 한다.

---

## 3. STOMP란?

STOMP는 Simple Text Oriented Messaging Protocol의 약자로, WebSocket 위에서 자주 사용되는 메시징 프로토콜이다.

WebSocket이 단순히 “메시지를 주고받는 통로”라면, STOMP는 그 통로 위에서 메시지를 어떤 형식으로 보내고, 어디로 전달할지에 대한 규칙을 제공한다.

STOMP에는 다음과 같은 프레임이 있다.

```text
CONNECT
SUBSCRIBE
SEND
MESSAGE
ACK
NACK
DISCONNECT
```

채팅 서비스를 예로 들면 다음과 같은 구조를 만들 수 있다.

```text
클라이언트가 채팅방 구독
/sub/chat/room/1

클라이언트가 메시지 발행
/pub/chat/message

서버가 구독자에게 메시지 전달
/sub/chat/room/1
```

즉, STOMP를 사용하면 WebSocket 위에서 pub/sub 구조를 비교적 쉽게 만들 수 있다.

Spring에서 STOMP를 사용할 때 흔히 보는 구조는 다음과 같다.

```text
클라이언트
  ↓ WebSocket 연결
STOMP frame
  ↓
/pub/chat/message 로 메시지 발행
/sub/chat/room/{roomId} 구독
  ↓
서버가 해당 방 구독자에게 메시지 전달
```

서버에서는 다음과 같이 메시지를 받을 수 있다.

```java
@MessageMapping("/chat/message")
public void send(ChatMessageRequest request) {
    chatService.send(request);
}
```

그리고 특정 채팅방을 구독 중인 클라이언트들에게 메시지를 보낼 수 있다.

```java
messagingTemplate.convertAndSend(
    "/sub/chat/room/" + roomId,
    response
);
```

이처럼 STOMP를 사용하면 직접 세션을 모두 관리하고, 방별 연결 목록을 직접 구성하는 부담을 줄일 수 있다.

---

## 4. STOMP의 장점

STOMP의 가장 큰 장점은 구조가 명확하다는 점이다.

보통 다음과 같이 역할을 나눈다.

```text
/pub = 클라이언트가 서버로 메시지를 보내는 경로
/sub = 클라이언트가 서버 메시지를 구독하는 경로
```

예를 들어 채팅에서는 다음과 같은 destination 구조를 만들 수 있다.

```text
/pub/chat/message
→ 사용자가 채팅 메시지 전송

/sub/chat/room/10
→ 10번 채팅방 메시지 구독

/user/queue/notification
→ 특정 사용자 개인 알림 수신
```

STOMP를 사용하면 다음과 같은 장점이 있다.

- pub/sub 구조를 쉽게 만들 수 있다.
    
- 채팅방 구독 구조가 명확해진다.
    
- Spring에서 지원이 좋다.
    
- 구현 난이도가 비교적 낮다.
    
- 일반적인 웹 채팅이나 실시간 알림 기능에 적합하다.
    
- RabbitMQ 같은 외부 STOMP broker와 연동할 수 있다.
    

그래서 일반적인 웹 채팅이나 포트폴리오 수준의 실시간 채팅을 구현할 때는 WebSocket + STOMP 조합이 가장 현실적인 선택이 될 수 있다.

---

## 5. STOMP를 쓰면 채팅 안정성이 자동으로 보장될까?

그렇지는 않다.

STOMP가 도와주는 것은 주로 다음 영역이다.

```text
- 구독
- 발행
- destination 기반 라우팅
- 메시지 프레임 구조
```

하지만 실제 메신저 서비스에 필요한 안정성은 별도로 설계해야 한다.

예를 들어 다음과 같은 기능은 STOMP만으로 해결되지 않는다.

```text
- 메시지 DB 저장
- messageId 생성
- clientMessageId 기반 중복 방지
- 방 안 메시지 순서 보장
- 재연결 시 누락 메시지 조회
- 읽음 처리
- 오프라인 사용자 푸시 알림
- 서버 다중화 시 메시지 전달
```

즉, STOMP는 채팅 구현을 편하게 해주는 메시징 규칙이지, 완성형 메신저 시스템은 아니다.

채팅 서비스의 안정성은 STOMP 사용 여부보다 메시지 저장, 중복 방지, 순서 보장, 재연결 복구 같은 설계에서 결정된다.

---

## 6. RSocket이란?

RSocket은 TCP, WebSocket 같은 전송 계층 위에서 동작하는 애플리케이션 메시징 프로토콜이다.

중요한 점은 RSocket이 WebSocket과 같은 레벨이 아니라는 것이다.

```text
WebSocket = 양방향 통신 연결을 제공하는 프로토콜

RSocket = TCP 또는 WebSocket 위에서 동작하는 애플리케이션 메시징 프로토콜
```

즉, RSocket은 WebSocket을 대체한다기보다는 WebSocket 위에서도 동작할 수 있는 상위 프로토콜에 가깝다.

구조는 다음과 같이 볼 수 있다.

```text
RSocket over TCP
```

또는 브라우저 환경에서는 다음과 같은 형태가 될 수 있다.

```text
RSocket over WebSocket
```

RSocket은 단순히 메시지를 보내고 받는 것보다 더 다양한 통신 모델을 제공한다.

대표적으로 다음과 같은 통신 모델이 있다.

```text
Request-Response
→ 요청 하나, 응답 하나

Fire-and-Forget
→ 메시지를 보내기만 하고 응답은 기대하지 않음

Request-Stream
→ 요청 하나에 대해 여러 개의 응답 스트림을 받음

Channel
→ 클라이언트와 서버가 양방향 스트림을 계속 주고받음
```

이런 특성 때문에 RSocket은 단순한 웹 채팅보다는 양방향 스트리밍, 서버 간 통신, 실시간 데이터 스트림 처리 같은 상황에서 더 잘 어울린다.

---

## 7. WebSocket, STOMP, RSocket 비교

세 기술을 비교하면 다음과 같다.

|구분|WebSocket|STOMP|RSocket|
|---|---|---|---|
|레벨|통신 프로토콜|메시징 서브프로토콜|애플리케이션 메시징 프로토콜|
|주 역할|양방향 연결 제공|pub/sub 메시징 규칙 제공|양방향 스트림, request-stream, backpressure 지원|
|단독 사용|가능|보통 WebSocket 위에서 사용|TCP 또는 WebSocket 위에서 사용|
|채팅 구현 난이도|직접 구현할 것이 많음|쉬운 편|어려운 편|
|방 구독 구조|직접 설계 필요|구현하기 쉬움|직접 모델링 필요|
|브라우저 호환성|좋음|좋음|WebSocket transport 필요|
|일반 웹 채팅 적합성|가능하지만 구현 부담 있음|적합|과할 수 있음|
|서버 간 통신 적합성|제한적|제한적|적합|
|대규모 스트리밍|직접 설계 필요|한계 있음|상대적으로 적합|

## 8. 채팅을 각각의 방식으로 구현하면?

### 8-1. 순수 WebSocket

순수 WebSocket만 사용하면 구조는 단순하다.

```text
클라이언트
  ↓
WebSocket
  ↓
서버
```

하지만 서버에서 직접 처리해야 하는 일이 많다.

```text
- 어떤 사용자가 어떤 방에 들어왔는지 관리
- 방별 WebSocket 세션 목록 관리
- 메시지 broadcast 처리
- 연결 종료 시 세션 제거
- 개인 메시지 라우팅
- 재연결 처리
```

예를 들어 서버 내부에서 다음과 같은 자료구조를 직접 관리해야 할 수 있다.

```text
roomId -> sessions
userId -> session
sessionId -> userInfo
```

순수 WebSocket 방식은 자유도가 높지만, 그만큼 구현 부담도 크다.

---

### 8-2. WebSocket + STOMP

WebSocket 위에 STOMP를 얹으면 채팅방 구독과 메시지 발행 구조가 명확해진다.

```text
클라이언트
  ↓
WebSocket
  ↓
STOMP
  ↓
서버
```

클라이언트는 특정 채팅방을 구독하고, 메시지를 발행한다.

```text
SEND /pub/chat/message
SUBSCRIBE /sub/chat/room/{roomId}
```

일반적인 웹 채팅, 포트폴리오, 사내 메신저, 실시간 알림 기능에는 이 구조가 가장 현실적이다.

추천 구조는 다음과 같다.

```text
WebSocket 연결
+ STOMP destination
+ DB 메시지 저장
+ clientMessageId 중복 방지
+ message sequence
+ 재연결 시 누락 메시지 조회
+ 오프라인 FCM/APNs 알림
```

STOMP는 채팅 구조를 단순하게 만들어주지만, 메시지 저장과 안정성 보장은 여전히 애플리케이션에서 설계해야 한다.

---

### 8-3. RSocket

RSocket은 다음과 같은 방식으로 사용할 수 있다.

```text
클라이언트
  ↓
RSocket over WebSocket
  ↓
서버
```

또는 서버 간 통신에서는 다음과 같은 구조도 가능하다.

```text
서버 A
  ↓
RSocket over TCP
  ↓
서버 B
```

RSocket은 단순 채팅보다 다음과 같은 상황에 더 어울린다.

```text
- 양방향 스트림이 중요하다.
- 실시간 데이터 스트림이 많다.
- request-stream 구조가 필요하다.
- backpressure가 중요하다.
- 서버 간 통신도 같은 방식으로 묶고 싶다.
```

단순히 사용자가 메시지를 보내고, 같은 방의 사용자들이 메시지를 받는 일반적인 채팅이라면 RSocket은 과한 선택이 될 수 있다.

---

## 9. 대부분의 메신저는 어떤 방식을 사용할까?

대형 메신저가 내부적으로 반드시 STOMP나 RSocket을 사용한다고 보기는 어렵다.

오히려 대부분의 대형 메신저는 자체 프로토콜이나 장기 연결 기반 구조를 사용하고, 그 위에 메시지 저장소, ACK, 재전송, 푸시 알림 등을 조합한다.

일반화하면 다음과 같은 구조에 가깝다.

```text
클라이언트
  ↓
WebSocket / TCP / 자체 장기 연결 프로토콜
  ↓
실시간 게이트웨이 서버
  ↓
내부 메시지 큐 / 이벤트 시스템
  ↓
채팅 서버 / 메시지 저장소 / 알림 서버
  ↓
DB 저장 + 푸시 알림
```

즉 대형 메신저의 핵심은 단순히 WebSocket을 사용하느냐가 아니다.

중요한 것은 다음과 같은 요소다.

```text
- 장기 연결 기반 실시간 통신
- 메시지 저장소
- ACK
- 재전송
- 중복 방지
- 순서 보장
- 푸시 알림
```

브라우저나 데스크톱 클라이언트에서는 WebSocket 계열을 많이 사용할 수 있지만, 모바일 앱이나 대규모 시스템에서는 자체 프로토콜, TCP 기반 장기 연결, HTTP API, 푸시 알림을 함께 사용하는 경우가 많다.

---

## 10. 채팅 안정성에서 진짜 중요한 것

WebSocket, STOMP, RSocket 중 어떤 것을 사용하더라도 채팅 안정성은 결국 다음 설계에서 갈린다.

```text
1. 메시지를 DB에 먼저 저장하는가?
2. 서버가 messageId를 발급하는가?
3. 클라이언트가 clientMessageId를 보내 중복 요청을 막는가?
4. 방마다 sequence를 부여해 메시지 순서를 보장하는가?
5. 클라이언트가 마지막으로 받은 sequence 이후 메시지를 재조회할 수 있는가?
6. WebSocket 연결이 끊겼다가 재연결되면 누락 메시지를 복구하는가?
7. 온라인 사용자와 오프라인 사용자를 구분하는가?
8. 오프라인 사용자에게 FCM/APNs 푸시를 보내는가?
9. 서버가 여러 대일 때 메시지를 어떻게 공유하는가?
10. 느린 클라이언트 때문에 전체 방 전송이 막히지 않게 하는가?
```

결국 다음과 같이 정리할 수 있다.

```text
WebSocket = 실시간 통신 수단

STOMP = 채팅 pub/sub 구현을 편하게 해주는 규칙

RSocket = 더 고급 스트리밍과 양방향 메시징을 위한 프로토콜
```

채팅 안정성은 프로토콜 선택보다 메시지 보장 설계에 더 크게 좌우된다.

---

## 11. 실무적인 선택 기준

### 일반적인 웹 채팅

일반적인 웹 채팅이라면 다음 조합이 가장 현실적이다.

```text
WebSocket + STOMP
```

이유는 다음과 같다.

```text
- 구현이 쉽다.
- Spring 지원이 좋다.
- 채팅방 구독 구조가 명확하다.
- 포트폴리오로 설명하기 좋다.
- 일반적인 채팅 서비스 구조를 보여주기에 충분하다.
```

---

### 직접 세밀하게 제어하고 싶은 채팅

세션 관리와 메시지 라우팅을 모두 직접 제어하고 싶다면 순수 WebSocket을 사용할 수도 있다.

```text
순수 WebSocket
```

하지만 직접 구현해야 할 것이 많다.

```text
- 세션 관리
- 채팅방 관리
- broadcast
- ACK
- 재전송
- 에러 처리
```

학습 목적이라면 좋은 선택이 될 수 있지만, 빠르게 안정적인 채팅을 구현하는 목적이라면 STOMP가 더 현실적이다.

---

### 고급 양방향 스트리밍 또는 서버 간 통신

다음과 같은 경우에는 RSocket을 고려할 수 있다.

```text
RSocket
```

적합한 상황은 다음과 같다.

```text
- request-stream이 필요하다.
- 양방향 channel이 중요하다.
- backpressure가 중요하다.
- 서버 간 실시간 스트림 통신이 많다.
```

하지만 일반적인 채팅 서비스에서 RSocket을 사용한다면 다음 질문에 답할 수 있어야 한다.

```text
“WebSocket + STOMP로 충분한데 왜 RSocket을 선택했는가?”

“RSocket의 backpressure가 실제 채팅에서 어떤 문제를 해결했는가?”

“브라우저 클라이언트 호환성은 어떻게 처리했는가?”

“운영 복잡도는 어떻게 감당할 것인가?”
```

이 질문에 명확한 답이 없다면 RSocket은 과한 기술 선택으로 보일 수 있다.

---

## 12. 최종 정리

마지막으로 세 기술을 한 문장씩 정리하면 다음과 같다.

```text
WebSocket
= 실시간 양방향 통신을 위한 연결 프로토콜

STOMP
= WebSocket 위에서 pub/sub 채팅 구조를 쉽게 만들기 위한 메시징 프로토콜

RSocket
= TCP 또는 WebSocket 위에서 동작하는 고급 애플리케이션 메시징 프로토콜
```

일반적인 채팅이라면 다음 구조가 가장 현실적이다.

```text
WebSocket + STOMP
```

고급 스트리밍이나 서버 간 양방향 통신까지 고려해야 한다면 RSocket을 검토할 수 있다.

```text
RSocket over TCP
RSocket over WebSocket
```

대형 메신저는 단순히 WebSocket 하나로 동작하지 않는다. 보통 WebSocket 또는 유사한 장기 연결 구조 위에 자체 프로토콜, 메시지 저장소, ACK, 재전송, 중복 방지, 순서 보장, 푸시 알림을 함께 조합한다.

결론적으로 채팅 서비스를 만들 때 중요한 것은 “어떤 프로토콜을 쓰는가”만이 아니다.

더 중요한 것은 다음 질문에 답할 수 있는 설계다.

```text
메시지가 유실되지 않는가?
중복 전송을 막을 수 있는가?
순서를 보장할 수 있는가?
끊겼다가 다시 연결됐을 때 복구할 수 있는가?
오프라인 사용자에게 알림을 보낼 수 있는가?
서버가 여러 대일 때도 메시지를 전달할 수 있는가?
```

WebSocket은 실시간 통신의 통로이고, STOMP는 그 통로 위에서 채팅 구조를 쉽게 만드는 규칙이며, RSocket은 더 복잡한 양방향 스트리밍 요구사항을 처리하기 위한 고급 메시징 프로토콜이다.

따라서 일반적인 웹 채팅에서는 WebSocket + STOMP를 먼저 고려하고, 더 복잡한 스트리밍 요구사항이 명확할 때 RSocket을 검토하는 것이 현실적인 선택이다.
